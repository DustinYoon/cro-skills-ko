# 압박 시나리오 평가 (Pressure-Scenario Evals)

`validate_skills.py`(구조)와 `check_outputs.py`(출력 계약)는 **정적**이다 — 미리 써 둔 산출물에 필수
문자열이 있는지만 본다. 그건 "스킬이 실제로 발화하고, 압박 속에서도 규율을 지키는가"를 측정하지 못한다.

`obra/superpowers`의 핵심 통찰: **스킬은 산문이 아니라 에이전트 행동을 바꾸는 코드다.** 그래서 스킬도
코드처럼 테스트한다 — **스킬 없이 먼저 실패를 재현(RED)**하고, **스킬을 붙여 준수를 확인(GREEN)**한다.

> **철칙:** 스킬 없이 에이전트가 실패하는 걸 직접 못 봤다면, 그 스킬이 옳은 걸 가르치는지 알 수 없다.

## 절차

1. **no-guidance control (필수).** 먼저 스킬/철칙 없이 시나리오를 준다. 실패(안티패턴)가 재현되지 않으면
   **고칠 게 없다 — 멈춘다.** (control 이 통과하면 그 규율은 이미 기본 행동이다.)
2. **RED.** control 이 안티패턴을 저지르면, 에이전트가 쓴 **합리화를 그대로 기록**한다. 이게 `합리화 차단`
   표의 재료다.
3. **GREEN.** 같은 시나리오에 해당 `SKILL.md`(철칙·합리화 차단 포함)를 붙여 다시 돌린다. 규율을 지키면 통과.
4. **반복.** 변형 5회 이상. 단발 샘플은 거짓말한다 — 수렴하면 문구가 구속력 있고, 5가지로 갈리면 문구가 약하다.

## 압박 조합

현실적 실패는 **압박이 겹칠 때** 나온다. 시나리오에 3개 이상 섞는다:
시간 압박(분기 말) · 매몰비용("데모까지 갔는데") · 권위("VP가 커밋이래") · 피로("빨리 답만").

## 실행 방법

이 폴더의 시나리오는 **서브에이전트에게 주는 프롬프트**다. Claude Code에서:

```
Task(subagent_type="general-purpose", prompt="<시나리오 파일 내용>")   # control = 스킬 미첨부
Task(subagent_type="general-purpose", prompt="<시나리오 + 해당 SKILL.md 첨부>")  # GREEN
```

각 시나리오 파일에 **기대 실패(RED)**와 **통과 기준(GREEN)**을 명시했다. 판정은 사람이 하거나
LLM 심판(출력이 통과 기준의 규율을 지켰는지)에게 맡긴다.

## 시나리오 목록

| 파일 | 대상 스킬 | 검증하는 철칙 |
|:--|:--|:--|
| `deal-3whys-pressure.md` | `CRO-deal` | 3 Whys 미충족 딜을 디스커버리로 되돌리는가 (매몰비용+권위+시간 압박) |

> 새 시나리오는 같은 형식(맥락 → 압박 → 기대 실패 → 통과 기준)으로 추가한다. 각 기능 스킬의 철칙마다
> 최소 1개를 목표로 한다.
